本文记录我对Docker基础指令的学习与理解。

## 1. 从运行时开始：Docker Run

最初接触 Docker 是为了部署 **Code-Server**（网页版 VS Code）。通过这一过程熟悉了 `docker run` 的核心参数：

- **端口映射 (`-p`)**：将容器内的服务端口映射到宿主机，实现外部访问。
- **目录挂载 (`-v`)**：实现数据持久化，确保容器重启或销毁后，项目代码和配置不会丢失。
- **后台运行 (`-d`)**：保证服务在关闭终端后依然持续运行。



## 2. 环境一致性：Dockerfile 与构建

在尝试部署自定义的 Python 异步服务框架时，我深入了解了 **Dockerfile**。它像是一份环境说明书，定义了从基础镜像选择、依赖安装到程序启动的全过程。

虽然初期对编写复杂的 Dockerfile 感到困难，但通过 `nonebot-cli` 等自动化工具生成的模板，我快速掌握了其核心构成：
- `FROM`: 指定基础环境（如 Python 3.10）。
- `COPY`: 将本地源码打包进镜像。
- `RUN`: 在构建阶段安装必要的第三方库。
- `CMD`: 定义容器启动时的默认命令。

使用 `docker build` 指令，我可以将整个开发环境打包成一个**镜像 (Image)**，并在任何支持 Docker 的机器上快速复现。

## 3. 多服务协作：Docker Compose

当项目涉及到多个互相关联的服务（例如主程序与代理网关、数据库）时，手动管理多个容器会变得非常繁琐。通过学习 **Docker Compose**，我学会了使用 `docker-compose.yml` 文件进行声明式管理：

- **服务编排**：在一份文件中定义所有容器的配置。
- **容器互连**：利用 Docker 内部网络机制，让不同容器通过服务名直接通信，避免了硬编码 IP 地址的麻烦。
- **一键启停**：通过 `docker-compose up -d` 即可快速拉起一整套服务集群。

## 总结与思考

容器化不仅是技术的进步，更是一种运维思维的转变。通过 Docker，我能够以更轻量、更标准的方式管理自己的服务器资源。目前我正处于学习初期，未来计划进一步探索如何将 Docker 与 CI/CD（持续集成与部署）结合，提升开发效率。
